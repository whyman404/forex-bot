# Database Migration Runbook

**Owner:** Mnemosyne Rin
**Last updated:** 2026-06-15
**Audience:** backend engineers, SRE, on-call

This runbook covers every routine and incident operation for the forex-bot
PostgreSQL database. If a step is missing, write the missing step; do not
improvise in production.

---

## 0. TL;DR — common commands

```bash
# Always run from backend/ (where alembic.ini lives).
cd backend/

# Apply all pending migrations.
alembic upgrade head

# Roll back the last migration.
alembic downgrade -1

# Roll back to a specific revision.
alembic downgrade 0001

# Show current revision.
alembic current

# Show migration history.
alembic history --verbose

# Generate a new (autogenerated) migration.
alembic revision --autogenerate -m "add foo column to users"

# Generate an empty migration (data migration, raw SQL, etc).
alembic revision -m "backfill country code on legacy users"

# Show SQL Alembic would execute (no DB write).
alembic upgrade head --sql

# Seed an admin user.
python -m scripts.seed_admin --email admin@local --password changeme123

# Seed a sample backtest (dev only).
python -m scripts.seed_sample_backtest

# Monthly partition maintenance (cron'd in prod).
python -m scripts.maintain_partitions --ahead 3

# Phase 2 backfill (idempotent — re-runnable).
python -m scripts.backfill_phase2
python -m scripts.backfill_phase2 --dry-run
```

## 1. Apply migrations (upgrade)

### Pre-flight (production)

- Confirm a fresh backup completed within the last 2 hours (see Sec. 5).
- Confirm peak hour avoidance (no migrations during NY market session for
  gold strategies; no migrations during weekend BTC volume spikes if any).
- Confirm app is on a Git tag and the migration has been QA'd in staging
  for at least 24 h.
- Acquire deploy lock.

### Steps

1. Verify the target revision: `alembic heads` (should be 1 head).
2. Inspect the SQL: `alembic upgrade head --sql > /tmp/plan.sql && less /tmp/plan.sql`.
3. Apply: `alembic upgrade head`.
4. Verify: `alembic current`.
5. Smoke-test the app: `curl -fsS https://api.../healthz`.
6. Release the deploy lock.

### If it fails partway

- Alembic wraps each migration in a transaction (unless you use
  `op.execute_isolation_level=AUTOCOMMIT`). A failure rolls back atomically.
- If it leaves the alembic_version table out of sync (rare):
  `alembic stamp <revision_id>` to manually set the cursor.
- Open an incident; do NOT silently rerun until you know what failed.

## 2. Create a new migration

### Schema change

```bash
# 1. Modify SQLAlchemy models under app/models/.
# 2. Generate skeleton:
alembic revision --autogenerate -m "add stop_loss_pct to strategy_instances"
# 3. Open backend/alembic/versions/<new_id>_*.py
#    - Verify both upgrade() and downgrade() are correct.
#    - Add data backfill if the column is NOT NULL with no default.
# 4. Use expand-contract for breaking changes:
#       step 1: add new column nullable
#       step 2: backfill
#       step 3: ship code that reads either old or new
#       step 4: drop old column in a SEPARATE migration ≥1 release later
# 5. Run alembic upgrade head in dev/test.
# 6. Run alembic downgrade -1 in dev to prove rollback works.
# 7. Re-run upgrade. Commit.
```

### Data-only migration (seeding, backfill, lookup tables)

```bash
alembic revision -m "seed new strategy: mean_reversion"
# Inside the file, use op.execute(sa.text(...)) or conn.execute(...).
# ALWAYS make it idempotent (ON CONFLICT DO NOTHING / WHERE NOT EXISTS).
# ALWAYS write the inverse in downgrade().
```

### Naming convention

`backend/alembic/versions/NNNN_short_slug.py` — NNNN is a 4-digit
zero-padded counter (see `alembic.ini` `file_template`). The next migration
after `0002_seed_strategies` should be `0003_...`.

## 3. Downgrade

### When to downgrade

- A migration deployed to staging caused a regression and we want to revert
  before debugging.
- A failed deploy was rolled back and we need to re-run forward without
  duplicate work.

### When NOT to downgrade

- Production after the app has been writing new data using the new schema
  for more than a few minutes. Forward-fix with a new migration instead.
- Migrations that DROP columns / tables (the data is gone — downgrade
  cannot bring it back).

### Steps

```bash
# Revert one revision.
alembic downgrade -1

# Revert to specific revision.
alembic downgrade 0001

# Reset everything (DEV ONLY — wipes all data).
alembic downgrade base
```

### Special case: 0001_initial.py

`downgrade()` in `0001_initial.py` is DESTRUCTIVE — drops every table.
It is provided for dev resets. Do NOT run in prod.

## 4. Re-seed

The platform expects these seeds to exist:

1. **Strategy catalog** — handled by migration `0002_seed_strategies` (data
   migration, idempotent).
2. **Admin user** — handled by `scripts/seed_admin.py` (idempotent UPSERT).
3. **(Optional) Sample backtest** — `scripts/seed_sample_backtest.py`
   (dev/staging only).

### Full re-seed sequence (e.g., after a restore)

```bash
# 1. Apply schema migrations.
alembic upgrade head

# 2. Seed the admin (changes password from default; secret rotation runbook
#    is in docs/security/secret-rotation.md).
python -m scripts.seed_admin --email admin@local --password "$(openssl rand -base64 24)"

# 3. (dev/staging) sample data.
python -m scripts.seed_sample_backtest
```

Strategies are seeded automatically by 0002. If you need to add or change
a strategy, write a NEW migration (do not edit 0002 — it's signed by its
revision and pinned to that snapshot).

## 5. Backup + Restore

### Backup schedule

| Type | Frequency | Retention | Storage |
|------|-----------|-----------|---------|
| Logical (pg_dump) | hourly | 48 h | S3 bucket `forex-bot-pgbackup` |
| Logical (pg_dump) | daily | 30 d | S3 + Glacier |
| Physical (WAL archive) | continuous | 14 d | S3 bucket `forex-bot-wal` |
| PITR base backup | weekly | 90 d | S3 + Glacier |

RPO = 5 minutes (WAL ship interval). RTO = 1 hour (target).

### Backup command (manual)

```bash
pg_dump \
  --host="$PGHOST" \
  --username="$PGUSER" \
  --dbname=forex_bot \
  --format=custom \
  --file="/backups/forex_bot_$(date -u +%Y%m%dT%H%M%S).dump"
```

### Restore — point-in-time (PITR)

1. Provision a new instance (or scratch the broken one).
2. Restore the most recent base backup.
3. Replay WAL up to the target timestamp:
   ```
   recovery_target_time = '2026-06-15 14:32:00+00'
   recovery_target_action = 'promote'
   ```
4. Promote when WAL replay reaches target.
5. Run `scripts/maintain_partitions.py --ahead 3 --skip-drop` to ensure
   the next month's partitions exist (recovery may skip the cron run).
6. Run `alembic current` — must match the revision the app expects.
7. Smoke test the app.
8. Cut DNS / connection string over.

### Restore — logical dump (full overwrite)

```bash
# DESTRUCTIVE. Only on a target DB you intend to wipe.
dropdb forex_bot
createdb forex_bot
pg_restore --dbname=forex_bot --jobs=4 --verbose /backups/forex_bot_*.dump
# Sanity: alembic current must show 0002 (or whatever head was at backup time).
```

### Restore drill — REQUIRED quarterly

Every 90 days, the on-call DBA must restore the most recent prod backup
to a staging environment and confirm:
- `alembic current` matches.
- A canary user can log in.
- A canary backtest completes successfully.
- The admin dashboard renders.

A backup that hasn't been restore-tested is not a backup.

## 6. Partition maintenance

`signals`, `trades`, and `audit_log` are RANGE-partitioned by month.

### Initial partitions

Created by `0001_initial.py`: m06 through m11 of 2026 (6 months ahead from
2026-06).

### Ongoing maintenance

```bash
python -m scripts.maintain_partitions --ahead 3
```

- Ensures the current month and the next 3 months exist.
- Drops partitions older than retention (signals 18 mo, trades 7 yr,
  audit_log 7 yr, email_outbox 6 mo, webhook_inbox 12 mo).
- Idempotent (uses pg_class existence check) — safe to rerun.

### Cron schedule (prod)

```cron
# Run on day 25 of each month at 02:00 UTC, then again on day 1 at 02:00 UTC
# (defence in depth: if the 25th run fails, the 1st catches it).
0 2 25 * *  cd /app && python -m scripts.maintain_partitions --ahead 3
0 2 1  * *  cd /app && python -m scripts.maintain_partitions --ahead 3
```

### What to do if a partition is missing

PostgreSQL routes rows to `*_default` partition. If you see rows in
`signals_default`, you missed a month — the cron failed.

To recover:
1. Run `maintain_partitions.py` immediately.
2. Move data from default to the right monthly partition:
   ```sql
   INSERT INTO signals_y2026_m12
       SELECT * FROM signals_default WHERE ts >= '2026-12-01' AND ts < '2027-01-01';
   DELETE FROM signals_default WHERE ts >= '2026-12-01' AND ts < '2027-01-01';
   ```
3. Investigate why the cron failed; add an alert if not already present.

## 7. Health + observability

### What to alert on

- Migration drift: app boot logs `Cannot find revision <X> in current head`
  → page on-call DBA.
- `signals_default` row count > 0 → partitioning missed a month.
- Replication lag > 30 s on replica.
- Disk free < 20%.
- `audit_log` write rate drops to 0 — possibly the trigger or app side broke.

### Dashboards

- Grafana: `Postgres / forex-bot` — TPS, lock waits, replication lag,
  partition row counts, slow queries.
- Sentry: `db.*` errors tagged by migration revision.

## 8. Troubleshooting

| Symptom | Likely cause | Fix |
|---------|--------------|-----|
| `relation "signals_y2026_mXX" does not exist` | Cron missed a month | Run `maintain_partitions.py` |
| `alembic upgrade head` hangs | Long-running transaction blocking DDL | `SELECT pid, query FROM pg_stat_activity WHERE state = 'active' AND query_start < now() - interval '1 minute'` then assess |
| `audit_log is append-only` exception | App is trying to UPDATE/DELETE audit_log | Bug — audit rows are immutable by design |
| Duplicate strategy code on seed | Someone edited 0002 instead of writing 0003 | Revert 0002 to original; add a new migration |
| `users_email_uniq` violation | Race during signup | Code-side check + retry with backoff |

## 8.5 Phase 2 deploy procedure (2026-06-15)

Phase 2 introduces migrations **0003** (additive schema) and **0004**
(seed plans), plus a backfill script.

### Pre-flight

- Fresh backup within 2 hours (Sec. 5).
- Apps using the affected tables (users, subscriptions, broker_accounts,
  strategy_instances) are on a release that tolerates the new nullable
  columns — i.e. expand phase.
- Stripe account has products created OR `stripe_setup.py` ready to run
  immediately after.

### Order of operations

```bash
# 1. Apply schema migration (additive — no downtime).
alembic upgrade 0003
alembic current   # must show 0003

# 2. Seed plans (idempotent).
alembic upgrade 0004
alembic current   # must show 0004

# 3. Backfill existing users (idempotent — can run during traffic).
python -m scripts.backfill_phase2 --dry-run   # inspect plan first
python -m scripts.backfill_phase2

# 4. (Outside DB scope) backend runs Stripe setup to populate
#    plans.stripe_product_id + plans.stripe_price_id.
python -m scripts.stripe_setup   # Atlas owns this

# 5. Ensure Phase-2 partitions are ahead for email_outbox + webhook_inbox.
python -m scripts.maintain_partitions --ahead 3
```

### Rollback

`alembic downgrade 0002` reverses both 0003 and 0004 cleanly because the
migration is additive — no data loss on the existing tables. New tables
(plans, stripe_events, ...) are DROPped; any rows in them are gone.

If 0004 has run AND the app started writing into `account_deletions`
or `plans`, do NOT downgrade — forward-fix.

### Post-deploy verification

```sql
-- Plans seeded?
SELECT code, display_name, price_cents, max_concurrent_live FROM plans ORDER BY sort_order;

-- New columns present?
SELECT column_name FROM information_schema.columns
 WHERE table_name = 'users' AND column_name IN
   ('onboarding_step','stripe_customer_id','consent_marketing_at');

-- Partitions exist for current + next 3 months?
SELECT relname FROM pg_class WHERE relname LIKE 'email_outbox_y%' ORDER BY relname;
SELECT relname FROM pg_class WHERE relname LIKE 'webhook_inbox_y%' ORDER BY relname;

-- Backfill complete?
SELECT count(*) FROM users WHERE onboarding_step = 0 AND email_verified_at IS NOT NULL;
-- ^ should be 0 after backfill (every verified user reaches at least step 1).

SELECT count(*) FROM subscriptions WHERE plan_id IS NULL;
-- ^ should be 0 after backfill (every legacy subscription mapped).
```
