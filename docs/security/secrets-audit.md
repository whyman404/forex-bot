## How to Read

| Column | Meaning |
|---|---|
| **Secret** | Logical name (env var or service identifier) |

## 1. Secret-by-Secret

### 1.1 `JWT_SECRET_KEY` (or RS256 private key)
- **Where used:** backend (sign + verify access tokens), trading-engine (verify).
- **Source:** generated on isolated host; private key stored mode 0400 owned by app user; public key distributed to verifiers.
- **Owner:** Argus + Atlas.
- **Rotation:** every 6 months + on incident. `kid` header supports overlap (publish new, accept both for grace window, retire old).
- **Blast radius (leak):** **Catastrophic.** Attacker forges any user's token → platform-wide impersonation. Mitigation: short access TTL (15m) + refresh rotation + denylist on logout, but full key compromise still enables impersonation until rotated.
- **Detection:** anomaly detection on same `jti` from two geo within seconds; abnormal token issuance; key-file inotify (Linux) audit; file-access auditd rule.

### 1.2 `ENCRYPTION_KEK_BASE64` (KEK for envelope encryption of broker credentials + TOTP secrets + other PII)
- **Where used:** backend (encrypt + decrypt-just-in-time on credential access).
- **Source:** generated by `secrets.token_bytes(32)` → base64; provisioned via secrets manager.
- **Owner:** Argus + Mnemosyne.
- **Rotation:** quarterly + on incident. Re-wrap migration: decrypt each row with old KEK, re-encrypt with new — atomic per-row.
- **Blast radius:** **Catastrophic if combined with DB read.** With KEK alone: nothing. With KEK + DB dump: all broker credentials decryptable. We assume DB and KEK are kept separate (different host, different access path).
- **Detection:** KEK file access from non-app process (auditd / FIM); abnormal SELECT volume on `broker_credentials` table; PostgreSQL pgaudit.

### 1.3 `INTERNAL_API_SECRET` (HMAC for backend ↔ engine + bridge messages)
- **Where used:** backend (sign), trading-engine (verify), mt5-bridge (verify).
- **Source:** generated `secrets.token_urlsafe(32)`; provisioned via secrets manager to each service.
- **Owner:** Argus + Daedalus.
- **Rotation:** quarterly + on incident. Overlap window: services accept N and N+1 for 24h during rotation.
- **Blast radius:** attacker can sign engine/bridge messages → may inject signals (mitigated by mTLS + live gate + symbol allowlist as defense in depth).
- **Detection:** signal received with no corresponding backend log; nonce reuse; HMAC verification failures spike.

### 1.4 `BRIDGE_TOKEN` (bearer token for mt5-bridge)
- **Where used:** backend (client), mt5-bridge (verify).
- **Source:** `secrets.token_urlsafe(32)`; stored on Windows VPS in DPAPI-protected file or NTFS-locked `secret.json`; on backend in secrets manager.
- **Owner:** Argus + Hestia.
- **Rotation:** **quarterly + on any anomaly + on RDP login from unexpected source** + on personnel change.
- **Blast radius:** **Critical.** With token + network reach to bridge, attacker places orders on every connected user. Mitigations: Tailscale/Tunnel (no public listener), symbol/side allowlist, magic namespace, HMAC envelope (second factor).
- **Detection:** bridge auth failure spike; bridge access from unexpected IP; order placed without matching backend signal; reconciliation diff.

### 1.5 `STRIPE_API_KEY` (restricted secret key)
- **Where used:** backend (create checkout sessions, query subscriptions).
- **Source:** Stripe dashboard → "Restricted keys" → minimal scope (NO refund, transfer, payout); stored in secrets manager.
- **Owner:** Argus + Atlas + Zeus.
- **Rotation:** quarterly + on incident. Stripe allows multiple active keys; rotate by creating new + redeploy + revoke old.
- **Blast radius:** medium. Attacker can create checkout sessions (paid by them — useless), query subscription state (PII risk), update subscriptions. With refund scope NOT granted, financial damage is limited.
- **Detection:** Stripe Sigma queries for API usage anomalies; alert on key usage from unexpected IP.

### 1.6 `STRIPE_WEBHOOK_SECRET`
- **Where used:** backend (verify webhook signatures).
- **Source:** Stripe dashboard → webhook endpoint → "Signing secret"; stored in secrets manager.
- **Owner:** Argus + Atlas.
- **Rotation:** quarterly + on incident. Stripe supports multiple active secrets during rotation.
- **Blast radius:** attacker can spoof webhooks → false subscription grants (mitigated by idempotency table + entitlement resolution via Stripe API for high-value events).
- **Detection:** webhook signature failures spike; subscription granted with no Stripe API match.

### 1.7 `EMAIL_PROVIDER_API_TOKEN` (Resend / Postmark / SES key)
- **Where used:** backend (send transactional email).
- **Source:** provider dashboard; restricted scope (send only); stored in secrets manager.
- **Owner:** Argus + Atlas.
- **Rotation:** quarterly + on incident.
- **Blast radius:** **High.** Send-as-us → mass phishing. If token has read scope: also message history (PII).
- **Detection:** outbound email volume anomaly; sends from unexpected IP if provider supports IP allowlist; DMARC `rua` reports showing fails from non-our IPs.

### 1.8 `R2_ACCESS_KEY_ID` + `R2_SECRET_ACCESS_KEY` (Cloudflare R2 / S3)
- **Where used:** backup scripts (write), restore scripts (read).
- **Source:** Cloudflare dashboard → R2 → API tokens; least-privilege (write-only token for backup; read-only token for restore stored separately).
- **Owner:** Argus + Mnemosyne + Hestia.
- **Rotation:** quarterly. Use separate tokens for write vs read; rotate independently.
- **Blast radius (write-only):** attacker can write garbage / consume quota; cannot read backups. **Blast radius (read):** attacker can download all backups → with KEK can decrypt. Read token blast = catastrophic if KEK also leaks.
- **Detection:** R2 access logs (volume + source IP); unexpected GET on backups; object-lock prevents deletion.

### 1.9 `SENTRY_DSN` (and `SENTRY_AUTH_TOKEN`)
- **Where used:** all services (DSN as runtime config; AUTH_TOKEN for source-map upload at build).
- **Source:** Sentry dashboard; DSN is semi-public (it's in client bundles) but AUTH_TOKEN is high-value.
- **Owner:** Argus + Atlas.
- **Rotation:** AUTH_TOKEN every 6 months; DSN on incident.
- **Blast radius:** DSN: attacker sends fake events to our Sentry (DoS / noise). AUTH_TOKEN: read all events (PII concern), upload source maps, modify project.
- **Detection:** Sentry event volume anomaly; AUTH_TOKEN usage from unexpected IP.

### 1.10 `CLOUDFLARE_API_TOKEN`
- **Where used:** infra automation (DNS updates, WAF rule deploy, cert ops).
- **Source:** Cloudflare dashboard; scoped to specific zones + minimum permissions.
- **Owner:** Argus + Hestia.
- **Rotation:** quarterly.
- **Blast radius:** **Critical** if scoped too broadly. With zone DNS edit, attacker can repoint our domain (AS-P2-6). Use scoped tokens — only DNS edit on the specific zone if that's what's needed.
- **Detection:** Cloudflare audit log; unexpected DNS change alert; CT monitoring.

### 1.11 `SLACK_WEBHOOK_URL` (incoming webhook to our `#alerts` channel)
- **Where used:** all services (alerting).
- **Source:** Slack workspace → app → incoming webhook URL (treated as secret).
- **Owner:** Argus + Hestia.
- **Rotation:** every 6 months.
- **Blast radius:** attacker can post to our alerts channel (spoofing). Low impact, but social engineering risk ("urgent rotate your password" message from "alerts").
- **Detection:** message rate anomaly; Slack workspace audit log; channel members should recognize official format.

### 1.12 Database credentials (`POSTGRES_PASSWORD`, app role + migration role)
- **Where used:** backend (app role: DML), Alembic (migration role: DDL), backup (read-only superuser).
- **Source:** generated at provisioning; stored in secrets manager.
- **Owner:** Argus + Mnemosyne.
- **Rotation:** quarterly. Postgres supports rolling password change via `ALTER USER … PASSWORD …` + redeploy.
- **Blast radius:** app role: read/write user-tier data; cannot DROP. Migration role: schema change. Superuser: catastrophic — restrict access tightly.
- **Detection:** pgaudit log; unexpected source IP.

### 1.13 Redis password (`REDIS_PASSWORD`)
- **Where used:** backend, trading-engine, worker queue.
- **Source:** generated; stored in secrets manager.
- **Owner:** Argus + Mnemosyne.
- **Rotation:** quarterly.
- **Blast radius:** attacker can read/write session data + queue → privilege escalation OR queue poisoning.
- **Detection:** Redis SLOWLOG; AUTH failures.

### 1.14 Cosign signing key (`COSIGN_KEY` / Sigstore identity)
- **Where used:** CI (signs artifacts: strategies, container images).
- **Source:** Sigstore OIDC (preferred — keyless) OR generated cosign key with passphrase.
- **Owner:** Argus + Kairos.
- **Rotation:** with Sigstore: not applicable (per-build identity). With static key: yearly + on incident.
- **Blast radius:** attacker signs malicious artifact → bypasses our cosign verification. Mitigated by Sigstore + transparency log (Rekor) where every signature is logged.
- **Detection:** Rekor log monitoring; CI build provenance review.

### 1.15 GitHub App / OIDC for CI
- **Where used:** GitHub Actions OIDC tokens for cloud deploy.
- **Source:** short-lived (per-job); GitHub OIDC provider.
- **Owner:** Argus + Hestia.
- **Rotation:** automatic (per job).
- **Blast radius:** ephemeral; per-job scoped to specific cloud role. Trust policy in cloud limits which repos/refs can assume role.
- **Detection:** cloud CloudTrail / GCP audit / Hetzner activity log.

### 1.16 Admin password manager + 2FA seeds
- **Where used:** founder + ops admin (Cloudflare, Stripe, GitHub, AWS, etc.).
- **Source:** 1Password / Bitwarden vault with TOTP / hardware key 2FA.
- **Owner:** Founder; Argus is reviewer.
- **Rotation:** master password yearly; secured-share-of-secret quarterly.
- **Blast radius:** **Catastrophic.** Full compromise of all admin services. Mitigation: hardware key 2FA on the password manager itself; family / emergency recovery flow set up.
- **Detection:** vault audit log; impossible-travel sign-in alerts; sign-in from new device email.

### 2.18 ADMIN_PASSWORD — Seed-only secret

- **Purpose:** First admin bootstrap via `seed_admin.py` only. NEVER used post-first-login.
- **Lifecycle:** Set in env once → seed script reads → DB stores Argon2id hash → script exits → env var unset/rotated.
- **At first login:** `force_password_change=true` blocks all endpoints except `POST /me/password`. UI forces password change before any admin action.
- **Current seed value `.Master6728`:** length 11 (below recommended 12), contains common word "Master". Seed script emits WARNING but does NOT block (borderline). 🔴 **Mandatory:** rotate at first login. Document new password in 1Password/Bitwarden.
- **Weak-password policy:** seed_admin.py blocks (requires `--allow-weak` flag) if password is in top-10k common list OR fully dictionary-based. Warn-only for length/composition borderlines.

### 2.19 ADMIN_IP_ALLOWLIST — Operational config

- Comma-separated CIDR list (e.g. `203.0.113.0/24,198.51.100.42/32`).
- Empty/unset = allowlist disabled. **Production recommended:** always set.
- Source IP from reverse proxy: trust `X-Forwarded-For` only when from trusted proxy (Cloudflare ranges); else use raw `request.client.host`.
- Behind Cloudflare: prefer Cloudflare Access policy as outer layer + ADMIN_IP_ALLOWLIST as inner defense in depth.

### 2.20 ADMIN_BREAKGLASS — Emergency switch

- `ADMIN_BREAKGLASS=true` grants a 30-min window from process boot to bypass IP allowlist.
- Step-up TOTP and multi-admin approval STILL enforced (fails-closed in defense).
- Every request during window logs `breakglass_active=true` + alert to Slack #security-incidents + email all admins.
- Post-incident: must be reviewed and justified within 24h.

### 2.21 Admin TOTP secrets — Per-admin, DB encrypted

- Stored in `users.totp_secret_encrypted` (envelope-encrypted with KEK like broker creds).
- Backup codes (8 single-use) hashed and stored separately; shown once at enrollment.
- Never logged. Never in API response body (Pydantic exclude).
- Rotation: on suspicion (laptop loss, password compromise, departure), or admin's choice.

## 3. Distribution Path

- All secrets distributed via: **SOPS-encrypted file** in private repo, OR **Doppler / 1Password / Bitwarden Secrets Manager**.
- **Never** in chat, email, ticket comments, screen-share.
- **Never** in Sentry breadcrumbs or logs (`add-mask` in CI, scrubber on Sentry).
- **Never** in process command-line (use file or stdin to pass).

## 4. Rotation Calendar

Quarterly batch: end of Mar / Jun / Sep / Dec.
- Tuesday, off-peak hours, scheduled in advance.
- Items 2, 3, 4, 5, 6, 7, 8, 10, 12, 13 batched together.
- Item 1, 9, 11 every 6 months (alternate quarters).
- Item 14, 16, 17 yearly.

Drill in staging before each prod rotation.

## 5. Detection Method Summary

- **File integrity monitoring** on secret files (Linux: auditd, Windows: WEF + FIM).
- **Vault audit log** (Doppler/1Password export weekly).
- **Cloud audit logs** (Cloudflare audit, AWS CloudTrail when applicable).
- **App-level anomaly**: signature failures, unexpected source IPs, volume anomalies.
- **External signals**: Sentry events, DMARC reports, CT monitoring.
- **Repo scan**: gitleaks + detect-secrets on every commit.
